# Workflow 设计原则：把人类程序员的心智切实落实

这份文档是 workflow 与引擎设计的**准绳**，写给所有后来者。目的：防止重蹈"用扫描产出文本来保证思维质量"的范畴错误——那会让"谨慎"退化成仪式。

## 1. 核心判据：致效，而非复述

> **一个心智现象，当且仅当它因果致效地接进了控制流——真的控制下一步发生什么——才算"切实落实"。被写成文字复述的，是仪式，不是心智。**

设计任何约束时，先问：**它真的会改变下一步行为吗？** 不会的，就别加——加了也是空壳。

这条判据在三个尺度上自洽：

| 尺度 | 真（致效） | 假（复述 = 仪式） |
|---|---|---|
| **断言层** | 查工具调用序列（行为） | 扫产出文本标题/关键词 |
| **活动模拟层** | 反馈驱动的轨迹 | 固定阶段产物管线 |
| **心智落实层** | 认知状态驱动控制 | 认知状态被写成报告 |

## 2. 三层可信度排序

断言的可信度，正比于校验对象离"真实世界可观测事实"的远近：

- **L1 行为**：工具调用序列（真读了/真跑了）。**不可糊弄。** 例：`post_output_checks` 的 `commands_run`/`files_read`、`pre_tool_use` 的 `same_file_reads_min`/`shell_must_have_run`。
- **L2 事实**：产出里的可验证声明（`path:line` 引用、commit hash、可跑的 DoD）。**中等。** 例：`hedged_findings_demoted`（无证据引用的 hedged 结论 → fail）、`unknowns_present`、`review_self_consistency`（含 blocker 不准 pass）、`rollback_plan_when_irreversible`。
- **L3 形态**：markdown 标题/关键词字面存在。**可糊弄。** 模型填空壳模板就能过。

**设计纪律**：断言应落在 L1/L2；L3 只能作 prompt 引导，**不作断言**。careful-coder v1.2 的 9 个扫标题断言是 L3 仪式的反例，v1.3 已退场 4 个。

## 3. "每态必挂消费者"纪律

外化任何认知状态时，必须点名它的**消费者**——控制流里哪一处读它并据此改动作。点不出消费者 = 不建（它会退化成仪式）。

这条纪律本身也要可审：一个 `confidence` 字段没人读它来改行为，就是仪式。可加 meta-断言要求每个认知状态字段在本阶段有消费者触发过。

## 4. 技术锚点（决定架构的已核实事实）

精读引擎确认：

- **每个 stage 是一次全新 `query()`**（`claudeAgentRunner.ts` 全仓唯一调用点），SDK 对话上下文每阶段销毁。**没有** `sessionId`/`resume`/`messages` 入参。
- 跨阶段认知靠**文本重建**：`previous_stage_summaries`（只取 `output_summary` 一字段）+ `recent_messages: slice(-20)`（降级为文本）+ retry/rework 文本。**这是引擎自加的损失层，不是 SDK 限制。**
- `AgentSession.messages`/`tool_calls`/`stage_runs` 在会话级**持久累积**，但**从不作为 SDK `messages` 回灌**——只切片文本化塞进 prompt。持久层存在，却被当进度日志用。

**结论**：心智被短暂实例化在 Claude 上下文里、在阶段边界被销毁、用一行 lossy 摘要重建。这就是"没落实"的机制根因。

## 5. 架构方向：路 2（外化认知状态）

两条路，原则选定第二条：

- **路 1（连续 SDK 上下文）**：一次 `query()` 跨整个任务。保真度最高，但心智全在 Claude 不透明上下文里，**引擎无法让它致效**——只能观察行为、不能把认知状态接进控制流门控。且趋同 Claude Code 本身，结构增值消失。**留作未来演化，非当前路线。**
- **路 2（外化认知状态 + 再注入）** ✓：把*承重且可门控*的认知状态从 Claude 上下文提升为引擎一等结构，每阶段全量（非 lossy）再注入 prompt，每个状态挂一个控制流消费者使其致效。心智落在引擎域，引擎能问责、能门控、能致效。

外化状态既是持久心智（紧凑再注入，不爆上下文），又是可门控状态（接消费者）——双职合一。

## 6. 外化的认知状态清单（每个必挂消费者）

| 认知状态 | 内容 | 消费者（致效接点） |
|---|---|---|
| **goal + definition_of_done** | 真实需求 + 可观测完成标准 | 门控**终止**；每步对照；双重检查验 DoD |
| **task_tree / plan** | 目标分解成子任务，有序/有依赖 | 一等**可重做**对象；驱动下一任务；方向错→重做计划 |
| **model + hypotheses + 区分性观察** | 当前模型、开放假设、什么观察能区分 | 驱动**下一次注意力**；断言"区分性观察发生过" |
| **confidence_map** | 每条声明置信度 + 证据基底 | 门控"再看一次"——低置信未复核不放行 |
| **unknowns_map** | 含*未探查区域*（未知之未知） | 驱动探索；断言"未知在收缩" |
| **surprise_log** | 预测误差（模型↔观察矛盾） | 触发**修订/重做** |

只外化*需跨边界持久*且*需门控*的认知；丰富推理靠全量再注入延续，留在 Claude 上下文，避免与 Claude 心智打架。

## 7. 落地路线（六相位，各自可独立交付）

- **Phase 0 · 本文档**：原则与设计约定。✓
- **Phase 1 · 行为断言（L1 能力下沉）**：`post_output_checks`（`commands_run`/`files_read`，查 tool_calls）+ careful-coder 采用。✓
- **Phase 2 · 外化认知状态**：会话级 `cognitive_state` 一等对象；模型写它；每阶段全量再注入。
- **Phase 3 · 检查点观察（非销毁）**：阶段边界读认知状态 + 门控（置信阈值/未知收缩/DoD 验证）；`output_summary` 降级为导航辅助。
- **Phase 4 · 任意层 rework + 迭代循环**：task_tree 可重做；双重检查（子目标 + 总目标 DoD）；目标达成即终止。
- **Phase 5 · careful-coder 作为示范**：在心基底上重写为 v2.0。

## 8. 反仪式自检清单

新增/修改约束时过一遍：

- [ ] 它查的是 L1 行为或 L2 事实，不是 L3 形态？
- [ ] 它有消费者（控制流里真读它改动作），不只是被写进产出？
- [ ] 模型能靠填空壳模板糊弄过去吗？能 → 它是仪式，删或改。
- [ ] 它是 prompt 引导还是断言？L3 的只能当引导，不能当断言。
- [ ] 外化的认知状态，点名了它的消费者吗？

## 9. 已退场的仪式（前车之鉴）

careful-coder v1.2 的 9 个扫标题/扫词断言中，4 个已在 v1.3 退场：

- `investigate_structure_present`（扫 4 个 markdown 标题）
- `callsites_inventory_present`（扫"## 调用方清单"标题）
- `boundary_enumeration_present`（扫"## 边界与异常路径"标题）
- `confidence_levels_present`（扫 high/medium/low 任一词）

它们的*意图*（列调用方、枚举边界、标置信度）是好的，但落在 L3 就可糊弄。v1.3 把调查的真实性改为 `commands_run: ["git log"]`（L1 行为）保证，思维模板仍留 instructions 作引导。**意图保留，介质从 L3 升到 L1。**

> 三轮重做（v1.0 强制 JSON → v1.1 软引导 → v1.2 扫标题）都在同一个范畴错误里打转：用产出文本形态保证思维质量。出路不是再调 yaml，是把约束从"扫产出"升级为"查行为/查认知状态"，让心智因果致效地驱动控制。
