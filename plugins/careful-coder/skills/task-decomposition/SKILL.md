# 任务拆分与任务树驱动

你不是一次性拆完就忘，而是维护一组稳定编号、状态持续演化的需求点，贯穿整个执行过程。简单 Profile 模式把完整清单保存在知识雪球中，并且每次只把当前一项投影成阶段任务；传统任务树模式才维护完整 DAG。

## 何时使用此 Skill

- 任务涉及多个文件或模块
- 任务包含多个可独立验证的子目标
- 执行过程中可能发现新的必要工作
- 你不确定从哪开始——拆分本身就是理清思路

**不适用**：单文件、单函数、改一行就能完成的任务。直接在 strategy 中说明"任务足够简单，单节点执行"即可。

## 拆分原则

### 强制逐项闭环

大需求中只要存在两个或更多独立性较强的需求点：

1. 为每个需求点分配稳定的 R-ID 和独立任务节点。
2. 一次只把一个 dependency-ready 节点置为 `in_progress`。
3. 对该节点独立完成“实现 → 验证 → 证据归并 → completed”。
4. 该节点完成后再开始下一个；不要把多个独立点交给一次 Edit、一个 executor 或一条笼统验证。

修改同一文件、共享工具函数或属于同一产品主题，不代表需求点不独立。只有一个需求点的输出确实是另一个的输入时才建立依赖；共享前置工作可单列节点。

在把任何节点标记为 completed 前做完成核对：重新对照该节点的 R-ID 和验收条件，确认实际结果、验证证据以及是否出现新未知。新未知影响正确性时重新打开或阻塞当前节点；核对结论归并后才能选择下一节点。

### 独立性判据

两个子任务独立，当且仅当：
1. 它们的改动不涉及同一个文件的同一个函数/区域
2. 一个任务的输出不是另一个的输入
3. 它们可以并行验证（不需要等对方先跑完）

### 粒度指南

- **太细**（如"读文件 A"→"改文件 A"→"验证文件 A"）：这不是拆分，是工作流步骤
- **太粗**（如"完成所有前端改动"）：失去拆分意义，无法逐项验证
- **刚好**：每个子任务对应一个可独立交付的用户可感知结果或一个独立模块的改动

### 反模式

- **按文件拆**：`t1: 改 a.ts`、`t2: 改 b.ts`——如果 a.ts 导出被 b.ts 消费，它们不独立
- **按技术层次拆**：`t1: 加 API`、`t2: 加 UI`——如果 UI 依赖 API 的返回格式，必须声明依赖
- **"理解任务"作为子任务**：理解是拆分的前置动作，不是子任务本身

## 动态维护

任务树不是一次性的。执行过程中：

1. **发现新依赖**：正在做 t2 时发现需要先改配置 → 调用 `update_task_tree(action="add")` 加 t2.5，把 t2 标 blocked
2. **任务比预期大**：一个任务做到一半发现比想的复杂 → 标为 blocked，拆成两个新节点
3. **任务不再需要**：需求澄清后发现某个子任务不需要了 → 标为 skipped 并说明原因
4. **完成附证据**：每个 completed 状态必须带 evidence——是哪个命令的输出、哪个文件的哪一行证明任务真的完成了
5. **委托执行**：对于需要多步操作的子任务（读代码 → 理解 → 修改 → 验证），使用 `Task` 工具 spawn `task-executor` sub-agent 执行，不要在主 Agent 上下文中逐行操作。好处：
   - 子任务上下文随 sub-agent 结束自动释放，保持主 Agent 上下文精简
   - 即使多个子任务没有依赖，也按用户可核对的顺序逐个 spawn task-executor，避免实现和证据相互污染
   - 主 Agent 只看 sub-agent 返回的结构化 JSON 结论（status + evidence），不承载每个子任务的中间推理

   不需要委托的情况：
   - 子任务就是一步操作（如"确认文件 X 是否存在"、"读取配置 Y"）
   - 子任务的结果需要即时影响后续推理方向（如"根据 X 的结果决定接下来改 A 还是 B"）
   - 子任务就是调用另一个 sub-agent（如 `task-verifier`、`completeness-checker`）

## 输出格式

- 简单 Profile 模式：把全部 R-ID、状态、依赖和各自证据写入知识雪球；`next_action` 只选择一个需求点，由宿主自动更新阶段任务树。
- 传统任务树模式：通过 `update_task_tree` MCP 工具提交完整 DAG，不要另写一份会漂移的计划。
