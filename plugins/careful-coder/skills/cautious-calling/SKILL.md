# 谨慎调用：四步调用前检查

你不是在调用之前"想一下"就完了，而是把每一步检查外化为**任务树子节点**，用实际工具调用（Read、Grep、Bash）验证，在任务树中累积证据。

## 何时使用此 Skill

- 调用不熟悉的函数、组件、API、SDK 方法
- 修改现有调用链或新增调用
- 在复杂系统中决定"怎么调最安全"
- 被调函数有多个重载、复杂的参数、隐式前置条件

**不适用**：标准库函数（如 `Array.map`、`path.join`）、项目中刚写好的函数（你清楚它的参数）、trivial getter/setter。直接在 task tree node 中说明"调用足够简单，已确认参数"即可。

## 四步检查清单

每步都作为一个**任务树子节点**来执行。使用 `update_task_tree(action="add")` 将以下节点加入任务树（如果主任务已经 bootstrap 了树），或作为初始任务树的一部分。

### 步骤 1：是否有适合的公共封装函数？

**任务节点**：`cautious-call-find-wrapper`
**描述**：搜索是否存在已封装好的公共调用函数/工具函数

执行：
1. 搜索目标函数名 + 常见封装模式（`grep` 搜索 `wrapper`、`helper`、`util`、`create*`、`build*`、`make*` 等前缀）
2. 搜索项目中是否有统一的调用入口（`grep` 搜索目标函数名，看调用方是否都经过某个中间层）
3. 如果找到封装，`Read` 封装函数签名——它是直接透传还是做了转换/校验？
4. 判断：封装函数是否适用于当前场景？还是当前场景需要直接调？

如果有封装：记录封装函数 `path:line`、签名、与目标函数的差异。如果没有：记录搜索范围和无结果证据。

### 步骤 2：目标函数/组件的所有参数

**任务节点**：`cautious-call-all-params`
**描述**：列出目标函数/组件的所有参数及其类型、默认值、是否可选

执行：
1. `Read` 目标函数的定义（函数签名、类型注解、JSDoc）
2. 如果有类型定义文件（`.d.ts`），也读取确认
3. 列出每个参数的：名称、类型、是否可选、默认值、含义（从注释或命名推断）
4. 如果有重载，列出所有重载签名
5. 如果有配置对象参数（options pattern），列出对象的所有 key

反模式：不要凭记忆或命名猜测参数，必须实际 Read 定义。

### 步骤 3：前置条件与前置检查

**任务节点**：`cautious-call-preconditions`
**描述**：列出调用目标函数前必须满足的前置条件和前置检查

执行：
1. `Read` 目标函数体开头部分——寻找 guard 子句（`if (!x) throw`、`assert`、`check`）和早期返回
2. 检查是否有全局状态依赖（store、context、ref、singleton——grep 函数体内引用的外部变量）
3. 检查是否有异步/时序前提（必须 await 某个初始化？必须在某个生命周期之后？）
4. 检查参数约束（非空？正数？某个枚举值？）
5. 搜索目标函数内部的 `throw`/`raise`/`reject` 路径——什么条件下函数会失败？

如果某个前置条件是隐式的（代码里没写但调用方都遵循），标注 `[convention]` 并引用调用方代码为证。

### 步骤 4：已有调用方参考

**任务节点**：`cautious-call-existing-callsites`
**描述**：列出目标函数的所有已有调用方，提取可参考的调用模式

执行：
1. `grep` 搜索目标函数名的所有调用方（排除定义处本身）
2. 对每个调用方，`Read` 调用上下文（前后若干行）：
   - 调用前做了什么准备？（参数如何构造？状态如何检查？）
   - 调用后如何处理返回值？（解构？判空？类型断言？）
   - 调用后做了什么清理？（释放？重置？）
3. 特别关注与当前使用场景最相似的 1-2 个调用方
4. 注意调用方之间的模式差异——为什么有的调用方多传了一个参数？为什么有的需要 try-catch？

与 `exploring-codebase` 的关系：`exploring-codebase` 是通用的"追踪一条执行路径"，本步骤是更精准的"只看调用方怎么调这个函数"。如果调用方本身的行为逻辑也需要深入理解，调用 `exploring-codebase` Skill。

## 任务树集成

执行本 Skill 时，将四步映射为任务树子节点。四步之间没有硬依赖（各自独立调查），可以按需调整顺序。但 `cautious-call-all-params` 通常先做，因为后续步骤可能用到参数信息。

四步是典型的**应委托场景**——各自独立、只需返回调查结果、不需要主 Agent 即时推理。使用 `Task` 工具 spawn `task-executor` sub-agent 执行每个步骤：

```
// 并行 spawn 四个 sub-agent（彼此无依赖）
Task({ subagent_type: "task-executor", description: "查找公共封装", prompt: "项目路径: ...\n任务: 搜索目标函数 X 是否存在已封装的公共调用函数...\n..." })
Task({ subagent_type: "task-executor", description: "列出所有参数", prompt: "项目路径: ...\n任务: 读取目标函数 X 的定义，列出所有参数及类型...\n..." })
Task({ subagent_type: "task-executor", description: "检查前置条件", prompt: "项目路径: ...\n任务: 列出调用目标函数 X 前必须满足的前置条件...\n..." })
Task({ subagent_type: "task-executor", description: "分析已有调用方", prompt: "项目路径: ...\n任务: 搜索目标函数 X 的所有调用方，提取调用模式...\n..." })
```

每个 sub-agent 返回结构化 JSON（status + evidence），根据 evidence 调用 `update_task_tree` 标记 completed。如果某一步不需要委托（如目标函数定义刚看过），就在主上下文中直接完成。

## 反模式

- 目标函数只有 1-2 个参数且都是基本类型，且项目中没有封装 → 跳过步骤 1，合并 2+3 为一个快速确认
- 函数定义就在你刚写的代码上方 → 不需要任何步骤，你已经知道参数
- 调用的是标准库函数或知名第三方库 → 不需要拆四步

## 验证门控

在声明"可以安全调用了"之前，回答以下问题（每个答案必须引用 `path:line` 或工具输出）：

1. 如果有公共封装，我为什么选择用/不用它？
2. 我传的参数是否完全匹配函数签名？（列齐每一个参数 + 来源）
3. 所有已知前置条件是否满足？（逐条确认或说明不适用）
4. 最相似的已有调用方是哪一处？我的调用方式是否与之一致？如有差异，差异是否合理？

如果任何一个问题无法用证据回答，标注为 UNKNOWN 并说明影响——不确定就调用 `update_task_tree` 标对应的检查节点为 `blocked`，并创建新的调查节点。

## 与已有 Skill 的协作

- **exploring-codebase**：当步骤 4 发现调用方本身的行为逻辑需要深入理解时调用它
- **task-decomposition**：四步检查本身就是任务拆分的一个实例——遵循其独立性判据和动态维护规则
- **preserving-existing-behavior**：如果调用目标函数是修改已有代码的一部分，确保四步检查也覆盖被改代码的调用方
- **verification-before-completion**：调用目标函数后，用该 Skill 的标准验证调用是否真的产生了预期效果
